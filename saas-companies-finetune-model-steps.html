Cell 2: Load the Base Model (4-bit QLoRA)

We load Qwen3-0.6B quantized to 4-bit. Unsloth handles the quantization automatically via the -bnb-4bit model variant.

Cell 2 — Load Qwen3-0.6B (4-bit) 
from unsloth import FastLanguageModel
import torch

# Load the base Qwen3-0.6B model in 4-bit to save GPU memory.
# The bnb-4bit variant is pre-quantized by Unsloth for fast loading.
model, tokenizer = FastLanguageModel.from_pretrained(
    model_name = "unsloth/Qwen3-0.6B-bnb-4bit",
    max_seq_length = 2048,
    dtype = None,       # Auto-detect: float16 or bfloat16
    load_in_4bit = True  # QLoRA: dramatically cuts VRAM usage
)

print("✅ Model loaded! Memory used:", round(torch.cuda.memory_allocated() / 1e9, 2), "GB")

Why Qwen3-0.6B instead of Llama 3.1 8B?

For a domain-specific lookup task (retrieve company info by name), a small model is perfectly sufficient. Qwen3-0.6B loads in seconds, trains in ~10 minutes on 500 examples, and the GGUF export is only 639MB — trivial to run locally with Ollama on any modern laptop.

Cell 3: Add LoRA Adapters

Cell 3 — LoRA Config 
# Add LoRA adapters: instead of training the whole model,
# we inject small trainable matrices into key attention layers.
model = FastLanguageModel.get_peft_model(
    model,
    r = 16,           # LoRA rank — capacity of adapter matrices
    target_modules = [
        "q_proj", "k_proj", "v_proj", "o_proj"  # Attention projections
    ],
    lora_alpha = 16,   # Scaling factor (keep equal to r)
    lora_dropout = 0,   # 0 = Unsloth-optimized
    bias = "none",
)

# Show trainable parameter count
trainable = sum(p.numel() for p in model.parameters() if p.requires_grad)
total = sum(p.numel() for p in model.parameters())
print(f"✅ LoRA added. Training {trainable:,} / {total:,} params ({100*trainable/total:.2f}%)")

Cell 4: Load and Format the Dataset

Cell 4 — Load companies_dataset.json 
import json
from datasets import Dataset

# Load your uploaded dataset file (500 SaaS company records)
with open('companies_dataset.json', 'r') as f:
    raw_data = json.load(f)

print(f"✅ Loaded {len(raw_data)} company records")

# Define the prompt template the model will learn to follow
prompt_template = """Instruction: Provide information about the company.
Company Name: {}
Response: {}"""

def format_data(example):
    # Pack URL, Category, Description into one response string
    response_text = (
        f"URL: {example['URL']}, "
        f"Category: {example['Category']}, "
        f"Description: {example['Description']}"
    )
    return {"text": prompt_template.format(example['Name'], response_text)}

# Convert JSON list to HuggingFace Dataset and apply formatter
dataset = Dataset.from_list(raw_data).map(format_data)
print(f"✅ Dataset ready. Sample:\n{dataset[0]['text'][:300]}...")

Cell 5: Configure Trainer & Start Fine-Tuning

Cell 5 — SFTTrainer (500 steps) 
from trl import SFTTrainer
from transformers import TrainingArguments

trainer = SFTTrainer(
    model = model,
    tokenizer = tokenizer,
    train_dataset = dataset,
    dataset_text_field = "text",
    max_seq_length = 2048,
    args = TrainingArguments(
        per_device_train_batch_size = 2,   # Rows processed at once
        gradient_accumulation_steps = 4,   # Simulates batch_size = 8
        max_steps = 500,                    # Total training steps
        learning_rate = 2e-4,              # Learning rate
        fp16 = True,                        # Half-precision math (faster)
        logging_steps = 1,                  # Print loss every step
        output_dir = "outputs",
    ),
)

print("🚀 Starting fine-tuning...")
trainer_stats = trainer.train()
print(f"✅ Training complete! Final loss: {trainer_stats.training_loss:.4f}")

Training Hyperparameters Explained

ParameterValueWhy
max_steps500~4 passes over 500 examples. Enough to learn company lookups without overfitting.
learning_rate2e-4Standard for LoRA fine-tuning. Higher = learns faster but risks instability.
batch_size2 × 4Effective batch of 8. Gradient accumulation simulates larger batch on limited VRAM.
fp16TrueHalf-precision speeds up training and reduces memory on T4 GPU.
⚠️ Expected Loss Curve: You should see training loss drop from ~2.0–3.0 down to ~0.3–0.8 over 500 steps. If loss stays flat after 100 steps, the learning rate may be too low. If it bounces erratically, try reducing to 1e-4.

Understanding the Output Files

model_gguf_gguf/Qwen3-0.6B.Q8_0.gguf (~639MB)

The merged, quantized model file. This contains both the original Qwen3-0.6B weights AND your fine-tuned LoRA adapter baked in. Q8_0 quantization gives excellent quality at half the size of a 16-bit model.

model_gguf_gguf/Modelfile

Ollama's configuration file. It references the GGUF file and sets inference parameters (temperature, top_p). You'll use this to register your model with Ollama as saas-orgs.

saas_model_lora/ (adapter only)

The raw LoRA adapter (~50MB). Useful if you want to: continue training from this checkpoint, merge with a different base model, or share just the adapter on Hugging Face Hub.

Using the Ollama Python API

Python — query saas-orgs via API 
import requests

def lookup_company(company_name: str) -> str:
    prompt = f"Instruction: Provide information about the company.\nCompany Name: {company_name}\nResponse:"
    
    response = requests.post(
        "http://localhost:11434/api/generate",
        json={
            "model": "saas-orgs",
            "prompt": prompt,
            "stream": False
        }
    )
    return response.json()["response"]

# Example usage
result = lookup_company("LinkedIn")
print(result)

Next Fine-Tuning Iteration — Improvements

Based on the current model's output, here are concrete improvements to make in the next iteration:

1. Dataset Cleaning (High Priority)

Remove or fix the ~30–40 records where Name and URL don't match (e.g., "Google" with a law firm URL). These cause the model to generate hallucinated/mixed responses. Filter with:

import json, re

with open("companies_dataset.json") as f:
    data = json.load(f)

# Simple heuristic: filter records where name appears in URL domain
def is_likely_valid(record):
    name_slug = record['Name'].lower().replace(' ', '')[:6]
    return name_slug in record['URL'].lower() or len(record['Description']) > 100

clean_data = [r for r in data if is_likely_valid(r)]
print(f"Kept {len(clean_data)} / {len(data)} records after cleaning")

2. Richer Prompt Format (Include Type + Keywords)

The current template only uses URL, Category, and Description. Adding Type and Keywords gives the model more signal:

def format_data_v2(example):
    response_text = (
        f"URL: {example['URL']}\n"
        f"Type: {example.get('Type', 'saas')}\n"
        f"Category: {example['Category']}\n"
        f"Keywords: {example.get('Keywords', '')}\n"
        f"Description: {example['Description']}"
    )
    return {"text": prompt_template.format(example['Name'], response_text)}

3. Increase Training Steps (500 → 1000–2000)

With a cleaned dataset of ~460 records, 500 steps gives roughly 1–2 epochs. Increasing to 1000–2000 steps will allow more passes through the data, improving recall accuracy for less-frequent companies.

Instead of restarting from Qwen3-0.6B base, continue from your existing saas_model_lora adapter to preserve what was already learned:
